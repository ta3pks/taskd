# taskd v2 architecture

_Thinking document created 2026-04-13T21:55:18.919Z_

## Thoughts

### 2026-04-13T21:55:34.503Z - Thought
## Problem 1: Per-task isolation

Current: single input.md/output.md pair — concurrent claude tasks clobber each other.

**Fix: task-scoped directories.** Each task gets its own dir under the bridge:
```
claude-code-bridge/
  <task-id>/
    input.md
    output.md
```
_build_claude creates the dir using task.id. Cleanup handled by existing purge.

This is the simplest change — task.id is already unique, mkdir is cheap. No shared state to coordinate.

### 2026-04-13T21:55:48.641Z - Thought
## Problem 2: Multi-channel notifications

Current: hardcoded Telegram Bot API. What if Discord/Slack/other channels later?

**Fix: Notifier interface with pluggable backends.** But YAGNI — we only have Telegram now. 

**Pragmatic approach:** Keep Telegram Bot API as primary (it works, it's direct). Add a generic notify config that maps channel_type -> endpoint:
```
TASKD_NOTIFY_CONFIG={"telegram": {"bot_token": "...", "chat_id": "..."}}
```
Or simpler: just keep current env vars but namespace them. When Discord is needed, add TASKD_NOTIFY_DISCORD_WEBHOOK. Don't abstract until there are 2+ channels.

**Decision: Keep current Telegram-only approach. Refactor when second channel is actually needed.**

### 2026-04-13T21:56:06.406Z - Thought
## Problem 3: Natasha seeing results in her conversation

This is the hardest one. Options:
1. **Webhook** — tried, doesn't inject into Telegram session. Natasha processes via HTTP but response doesn't reach Nikos.
2. **Bot sends message, Natasha sees it** — Bot API sendMessage goes TO the user, but Natasha only sees messages FROM the user. Bot's own messages don't trigger the polling loop.
3. **Natasha polls taskd** — Add taskd check to her 15-min cron or heartbeat. When she sees a completed claude task, she reads output and relays. Downside: up to 15min delay.
4. **Natasha polls taskd more frequently** — Dedicated 1-min cron just for claude task completion.
5. **Accept current design** — taskd sends result directly to Telegram. Nikos sees it immediately. If he wants Natasha to elaborate, he tells her.

**Decision: Option 5 is actually fine for a personal assistant.** The result lands in Telegram fast. Natasha doesn't need to be in the loop — she submitted it, the result arrived. If Nikos wants follow-up, he asks Natasha who can read the output file.

### 2026-04-13T21:56:20.565Z - Thought
## Problem 4: Multi-agent support

What if multiple agents (not just Natasha) submit tasks? E.g. a coding agent, a research agent, a monitoring agent.

**Current:** taskd has no concept of 'submitter'. Tasks are anonymous.

**Fix: Add optional submitter field to Task.**
- `taskd add claude --submitter natasha "desc" prompt`
- Notification includes submitter so the right agent gets pinged
- Per-submitter notification routing (Natasha -> Telegram, coding-agent -> webhook, etc.)

**But again YAGNI.** There's only Natasha now. Adding a submitter field is cheap (one string field). Routing can come later.

**Decision: Add submitter field to Task dataclass (default: empty). Include in notification message. Don't build routing yet.**

### 2026-04-13T21:56:38.195Z - Conclusion
## Architecture Decision: Incremental, Not Big-Bang

**Do now (v2):**
1. Per-task directories for claude bridge files (task-id scoped)
2. Add submitter field to Task (future-proofing, cheap)
3. Keep Telegram Bot API notifications (working, direct)
4. Purge cleans up per-task bridge dirs

**Do when needed (v3+):**
- Multi-channel notification routing (when second channel exists)
- Agent-specific notification endpoints
- Natasha conversation injection (if polling delay becomes unacceptable)

**Don't do:**
- Abstract notification interfaces
- Message queues or event buses
- Webhook-based conversation injection (tried, doesn't work with ZeroClaw's architecture)

The current design is 90% right for a personal assistant. Per-task isolation is the only real bug. Everything else is premature.
